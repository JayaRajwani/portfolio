// sample lesson

Turn a vague request into a Codex task

Lesson objective

By the end of this lesson, the learner can write a Codex prompt that gives enough context for useful work and enough boundaries for safe verification.

The pattern

Use four blocks: goal, context, constraints, and done when. The point is not to write a longer prompt. The point is to remove ambiguity that would otherwise become agent guesswork.

Weak prompt

Improve the search page and make it better.

Codex-ready prompt

Goal:
Improve search results loading states and empty states.

Context:
Start with src/pages/search.tsx and existing components in src/components/search/.

Constraints:
Do not add a new UI library. Match existing styles. Keep changes small.

Done when:
There is a loading state, an empty state, and the existing search tests pass.
Run the focused test command and summarize the diff risks.

Exercise

Take one vague request from your real backlog. Rewrite it with the four-block pattern. Ask Codex to restate the task, name missing context, and propose a plan before editing.

Expected output

  • Codex identifies the right files before editing.
  • The plan includes a verification step.
  • The final answer names what changed and what could not be verified.
  • The diff is small enough for a human to review in one sitting.

Instructor note

Beginners usually think prompting is about clever wording. In practice, it is about task design: what should the agent know, what should it avoid, and how will we know it worked?

Common failure

If "done when" is missing, Codex may still produce a plausible change, but the learner has no reliable way to judge completion.

// advanced lab

Codex + Claude + Hermes reliability loop

This lab teaches a practical separation of responsibilities for deployed agentic systems. Codex builds inside the repo, Claude critiques assumptions and edge cases, and Hermes evaluates whether the workflow behaves reliably across a repeatable test set.

step_01

Build with Codex

Implement a small agentic workflow, add logging or traces, write a focused test, and document the tool contract.

step_02

Critique with Claude

Ask for architecture risks, missing edge cases, prompt ambiguity, unsafe assumptions, and failure modes that are not covered by tests.

step_03

Evaluate with Hermes

Create a golden set, run the workflow repeatedly, compare expected and actual behavior, and track regressions over time.

Artifact What good looks like Why it matters
Golden set 10-20 realistic cases with expected behavior and edge cases Prevents demo-driven confidence
Failure taxonomy Categories such as retrieval miss, wrong tool, bad citation, unsafe action, or vague refusal Makes errors discussable and fixable
Trace review Each failed case includes the input, tools called, model output, and final verdict Turns "it failed" into a debugging path
Release bar Clear pass/fail rule before the workflow is trusted Stops subjective approval from creeping in
